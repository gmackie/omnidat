# Data Model

OMNIDAT should keep its data model small and boring. The physical installation
is complicated enough; the records should be easy to inspect, print, and repair.

## Core Entities

```text
Account
PacketNamespace
TransportProfile
Terminal
Endpoint
Service
Vendor
CarrierCircuit
CallRecord
SessionRecord
PrintJob
FaxRecord
MediaTape
MediaRequest
Incident
```

## PacketNamespace

Represents a reserved Packet Clearing address range and its provisioning rule.
Namespaces let OMNIDAT distinguish core services, trusted partners, merchant
circuits, approved villages, open campsite apps, event operations, and
diagnostics without making every packet address look equally official.

```json
{
  "namespace_id": "open-campsite",
  "range_start": "020000",
  "range_end": "029999",
  "label": "Open campsite applications",
  "service_class": "OPEN",
  "provisioning": "self-service",
  "directory_status": "provisional",
  "description": "Self-service campsite applications and provisional public directory entries"
}
```

The canonical namespace seed file is
`data/packet-namespaces.sample.json`.

## TransportProfile

Represents a way to attach to Packet Clearing. Transports do not own service
identity; they carry terminal, radio, modem, hosted, or remote-node requests to
the same packet address space.

```json
{
  "transport_id": "meshcore-radio-pad",
  "name": "MeshCore Radio PAD",
  "kind": "radio-pad",
  "authority": "managed-field-access",
  "default_access": "PASSPORT",
  "description": "Managed OMNIDAT loaner and fixed-node radio access using compact Radio PAD commands"
}
```

The canonical transport seed file is
`data/transport-profiles.sample.json`.

## Account

Represents a person, operator, test identity, or in-world business customer.

```json
{
  "account_id": "ACCT-000128",
  "display_name": "MACKIE G",
  "access_class": "REGISTERED",
  "extensions": ["8800"],
  "packet_permissions": ["DIRECTORY", "MEDIA_CATALOG"],
  "created_at": "2028-07-01T10:00:00-07:00",
  "status": "active"
}
```

## Vendor

Represents a merchant, back-office tenant, or service owner on the OMNIDAT
carrier network.

```json
{
  "vendor_id": "VEND-NITEMARKT",
  "name": "NiteMarkt",
  "kind": "anchor-tenant",
  "contact_number": "8819",
  "status": "planned"
}
```

## CarrierCircuit

Represents a private access path for a terminal, proxy, ATM, BOH system, or
vendor service.

```json
{
  "circuit_id": "CKT-NM-WMS-01",
  "vendor_id": "VEND-NITEMARKT",
  "terminal_id": "TERM-NM-BOH-01",
  "service_address": "000020",
  "access_class": "REGISTERED",
  "status": "planned"
}
```

Access classes:

```text
PUBLIC       no registration needed
REGISTERED   account card or operator-created account
OPERATOR     service desk access
MAINTENANCE  hardware and recovery functions
```

## Service

Maps public dialed numbers to behavior.

```json
{
  "service_id": "packet-main",
  "name": "OMNIDAT Packet Clearing",
  "number": "8810",
  "route_class": "hunt",
  "endpoints": ["PAD-01", "PAD-02"],
  "maintenance_mode": false,
  "channel_limit": 4
}
```

## CallRecord

Generated by the PBX for every call leg.

```json
{
  "call_id": "CALL-20280701-000042",
  "started_at": "2028-07-01T10:05:14-07:00",
  "ended_at": "2028-07-01T10:07:52-07:00",
  "caller": "ShadyTel:1234",
  "called": "8810",
  "service_id": "packet-main",
  "endpoint_id": "PAD-01",
  "disposition": "answered"
}
```

Dispositions:

```text
answered
busy
intercept
maintenance
failed
operator
abandoned
```

## Storage Recommendation

Use SQLite for V1. It is inspectable, portable, easy to back up, and adequate
for the expected event scale.

Suggested databases:

```text
omnidat-core.db       accounts, services, endpoints
omnidat-events.db     call/session/print/fax/media/incident records
omnidat-vault.db      media inventory and request queue
```

Export daily JSONL and print summaries so the event does not depend on one
database file being perfect.

## Event Ledger

The current event ledger is append-only JSONL:

```text
build/events.jsonl
```

Each line is one event:

```json
{
  "event_id": "EVT-20280701-000001",
  "type": "call.ended",
  "source": "pbx",
  "created_at": "2028-07-01T10:00:00-07:00",
  "payload": {
    "called": "8800",
    "disposition": "answered"
  }
}
```

Supported summary categories are:

```text
calls_completed
busy_or_intercepts
pad_sessions
bbs_sessions
faxes_received
print_jobs
media_requests
incidents
```

## Current Lab Database

The current repo tooling builds one combined lab database:

```sh
./scripts/build-db
```

Default output:

```text
build/omnidat.db
```

This is intentionally simpler than the eventual split-database plan. It gives
the lab build one inspectable data spine while the PBX, Packet Clearing,
Document Services, and Media Vault interfaces are still being proved.
